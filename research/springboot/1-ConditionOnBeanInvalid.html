# 记一次ConditionOnBean失效问题

# 1.自动配置类失效

我将 lc-xxx-starter 组件（某熔断限流组件），整合到 xxx-demo 项目中,

然后打开限流开关测试限流，发现限流不生效

debug 了埋点，未进入

初步推断 AutoConfiguration 没有注入到容器

如下是 lc-xxx-starter 的自动装配方式（保密需要，省略了非关键代码）:


@Configuration
@ConditionalOnClass(...)
@ConditionalOnBean(Entry.class)
@AutoConfigureAfter(EntryAutoConfiguration.class)
@ConditionalOnProperty(...)
public class RestAutoConfiguration {

    @Bean
    public RestWebMvcHandler restWebMvcHandler(Entry entry) {
        return new RestWebMvcHandler(entry);
    }

}

然后 debug 了 @Bean 里面的方法，果然没有进入

通过 actuator 接口查看 condition 信息，发现 @ConditionalOnBean 未生效

# 2.ConditionalOnBean失效

debug 了一下 OnBeanCondition ,确实返回了 false

表示这个配置类注入的时候,Entry 还没有注册到 BeanDefinion 中

又 debug 了一下 Entry 的创建方法


@Configuration
@EnableConfigurationProperties(Properties.class)
@ConditionalOnProperty(xxx)
public class EntryAutoConfiguration {

    @Bean
    @ConditionalOnMissingBean(Entry.class)
    public Entry entry(...) {
        return ...;
    }

}

Entry 进入了创建方法,说明 Entry 有在 BeanDefinion 中注册

# 3.AutoConfigureAfter失效

这就很奇怪了,推测了可能是顺序问题：

RestAutoConfiguration 先与 EntryAutoConfiguration 加载

如果是这样,表示 @AutoConfigureAfter 注解失效了

然后通过 debug 查看了配置类的加载顺序

验证了我的猜想,加载顺序确实乱了

# 4.Configuration加载顺序异常

debug 跟进了 AutoConfigurationSorter 相关代码,发现排序是正常工作的

然后 debug 最终加载结果,顺序却乱了

看了下排序数据结构是 LinkedHashMap

推测可能是 Configuration 被重复加载,导致顺序异常

通过 debug 验证了这个推测，Configuration 先被加载进去了，而后又被加载了一次

# 5.哪里在重复加载 EntryAutoConfiguration?

通过 debug 进入第二次加载 EntryAutoConfiguration 的断点

翻了一下调用栈,找相关类 XXXStarterAutoConfigure


@Configuration
@ComponentScan(basePackages = {
        "com.xxx",
        "com.xxx.core",
        "com.xxx.sdk",
})
public class XXXStarterAutoConfigure {
}

真相大白了,这里通过 ComponentScan 扫描了 com.xxx 下所有 bean

导致 EntryAutoConfiguration 被重复加载

重复加载导致顺序异常,最终导致 ConditionalOnBean 失效

# 6. 解决这个问题

1.去掉 ComponentScan

2.调整基础包名，不让它被扫描到
